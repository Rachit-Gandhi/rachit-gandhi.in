# Clarify Before Code: How I Use a Readiness Gate for AI-Assisted Engineering

Fast code generation is useful, but in production work the real bottleneck is usually not typing speed — it’s **decision quality**.

That’s why I built a custom Codex Skill Builder workflow that forces a readiness gate before implementation. The goal is simple: reduce avoidable rework by increasing clarity first.

## The core idea

Instead of "prompt → code → patch", the workflow is:

1. Clarify intent
2. Route to the correct codebase context
3. Score readiness
4. Only then allow mutating implementation

This sounds slower, but in practice it shortens total cycle time by cutting rewrite loops.

## What the gate actually checks

I score seven dimensions, each from `0` to `2`:

- Requirements and scope
- Project route and context confidence
- Current codebase integration points
- Constraints and non-goals
- Naming conventions (micro)
- Naming conventions (macro)
- Acceptance criteria and verification

Gate bands:

- `RED` (0–7): too ambiguous, stay in clarification mode
- `YELLOW` (8–11): partial clarity, resolve high-risk gaps
- `GREEN` (12–14): safe to execute implementation

I also keep hard guards: requirements, codebase route, and acceptance criteria cannot be weak even if total score looks acceptable.

## Functionalities that matter in real projects

### 1) Clarification rounds, not one giant questionnaire

I run short rounds of targeted questions so momentum stays high:
- what’s in scope vs out of scope,
- where this change belongs,
- what must stay backward-compatible,
- what proof will mark done.

### 2) Pre-green tool budget

Before gate is green, discovery is allowed but constrained:
- read-only checks only,
- fixed tool-call budget,
- no file mutations.

This prevents premature implementation while still allowing useful reconnaissance.

### 3) Context routing before coding

The workflow explicitly validates which repo/module should own the change. Misrouting is a silent source of wasted effort, so this check pays off quickly.

### 4) Naming decisions at two levels

I separate naming into:
- **micro**: symbols, functions, files,
- **macro**: modules, services, boundaries.

This improves consistency and makes reviews much cleaner.

### 5) Verification-first definition of done

Before coding starts, expected evidence is already defined:
- mandatory tests/checks,
- regression boundaries,
- user-visible behavior expectations.

So implementation happens against a concrete finish line, not fuzzy intent.

## A sample gate snapshot

```md
## Readiness Score
- Overall: 10/14 (YELLOW)
- Requirements: 2/2
- Route & Context: 1/2
- Codebase Integration: 1/2
- Constraints: 2/2
- Naming (Micro): 1/2
- Naming (Macro): 1/2
- Acceptance Criteria: 2/2
- Missing for GREEN: module ownership proof, integration boundary confirmation
```

This makes uncertainty visible, discussable, and fixable.

## Why I keep using this

This workflow consistently gives me:
- fewer assumption-driven bugs,
- fewer rewrite passes,
- better first-pass architecture decisions,
- cleaner handoffs and reviews.

So my conclusion is:

AI coding quality is heavily shaped by **when** we allow execution.
A good gate doesn’t reduce velocity — it protects it.
